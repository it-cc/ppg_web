## 文档定位与使用方式

本技术文档面向两个核心使用场景进行统一设计。第一类场景是路演PPT中的技术展示，需要在有限页面内清晰传达产品价值、技术可行性与落地能力。第二类场景是企划书中的详细技术说明，需要提供可核验的系统架构、算法路径、工程实现与风险控制依据。基于此，本文在结构上采用“先概览、后细化、再落地”的叙述方式，使同一份内容既可被提炼为展示级信息，也可直接作为方案级技术章节。

## 项目目标与价值主张

本项目构建了一个面向光电容积脉搏波信号的智能分析系统，目标是在标准Web环境中完成从数据接入、信号处理、特征计算、图形展示到智能报告生成的闭环流程。系统以快速、低门槛和可扩展为设计原则，支持CSV与JSON格式数据输入，在后端执行生理指标计算并在前端提供可解释的可视化结果，同时调用本地大模型生成结构化健康解读。该设计使系统能够在研发验证、医疗辅助评估、健康管理演示与产业合作沟通中承担“技术样机+产品原型”的双重角色。

## 总体技术架构

系统采用前后端分离架构，前端由Vue 3、Vite与TypeScript构成交互层，后端由FastAPI承载计算与服务编排。前端负责文件上传、参数配置、图表渲染和AI报告展示，后端负责信号解析、预处理、特征提取及流式智能分析输出。两端通过HTTP接口进行解耦通信，接口粒度与职责边界清晰，便于后续独立扩展。该架构的核心优势在于迭代效率高、部署路径清晰、维护成本可控，并且能够平滑迁移到容器化与分布式运行环境。

## 关键模块设计

前端分析模块围绕单页工作台设计，提供数据源接入、算法参数配置、实时状态反馈与多视图切换能力。时域图用于呈现原始波形与滤波后波形的差异，频域图用于展示功率谱分布，峰值检测图用于直观标定关键形态点。该可视化组合使技术评审方能够快速判断信号质量、处理效果与特征可信度，从而降低纯文本报告在沟通中的理解成本。

后端分析模块由接口层、处理层和智能服务层构成。接口层负责接收数据并进行格式校验，处理层完成信号处理与生理特征提取，智能服务层基于特征结果输出自然语言报告。系统当前包含面向缓冲信号的分析接口、面向文件上传的分析接口以及面向特征输入的AI分析接口。接口职责单一且命名语义明确，能够支持前端交互、自动化测试和第三方系统集成。

## 数据流与处理流程

系统的数据流从“输入信号”进入“标准化处理”，再进入“特征计算”和“智能解释”，最终回到“可视化与文本输出”。在输入阶段，后端从CSV或JSON中读取首列有效数值并构建时序序列。在处理阶段，系统执行滤波与有效片段裁剪，控制返回数据长度以确保前端渲染稳定性。在计算阶段，系统生成心率HR、血氧估计SpO2、呼吸频率RR与心率变异性HRV_SDNN等核心特征。在输出阶段，前端将结构化结果映射到统计卡片与图表组件，同时将特征向量发送给AI服务生成报告文本。

该流程可抽象为如下映射关系，其中 $x(t)$ 表示原始信号，$\hat{x}(t)$ 表示处理后信号，$F$ 表示特征向量，$R$ 表示报告结果。

$$
F = \Phi(\hat{x}(t)) = \Phi(\Psi(x(t)))
$$

$$
R = \Gamma(F)
$$

式中，$\Psi(\cdot)$ 对应信号预处理算子，$\Phi(\cdot)$ 对应生理特征提取算子，$\Gamma(\cdot)$ 对应智能语言生成算子。该分层映射保证了每一层的可替换性，便于后续引入更高精度模型或领域规则。

## 核心接口与实现示意

后端接口设计遵循“功能最小闭环”原则。分析接口能够处理上传文件并返回可直接渲染的数据结构，智能分析接口则接收特征对象并以流式形式返回结果文本。以下示意代码反映了系统中的关键服务编排逻辑。

```python
@app.post("/api/analyze")
async def analyze_ppg(file: UploadFile = File(...)):
    contents = await file.read()
    if file.filename.endswith('.csv'):
        df = pd.read_csv(io.StringIO(contents.decode('utf-8')))
    elif file.filename.endswith('.json'):
        df = pd.read_json(io.BytesIO(contents))
    else:
        return {"error": "Unsupported file format"}

    raw_signal = df.iloc[:, 0].dropna().tolist()
    result = process_buffer(raw_signal, fs=100)
    return result
```

该实现通过统一入口屏蔽了不同文件格式差异，在不增加前端复杂度的前提下实现了多格式兼容。接口内部将格式解析、异常返回与核心分析流程进行顺序编排，使错误路径与成功路径同样可追踪。对于路演场景而言，这种实现方式能够直观体现系统的工程完整度；对于企划书场景而言，这种结构则体现了可维护性与扩展性。

## 智能分析能力与可解释性

系统的AI能力并非替代传统信号分析，而是建立在可量化特征之上的语义增强层。模型输入是明确的生理指标对象，输出是可阅读的健康分析文本。由于特征来源可回溯到波形与频谱图，报告中的结论具备可解释路径，评审方可以通过“指标值—图表证据—文本结论”的链路进行交叉验证。这一机制能够有效降低“黑盒结论”带来的信任壁垒，并提升路演过程中的技术说服力。

## 工程化部署与运行条件

系统可在标准开发环境中快速部署。后端依赖FastAPI与科学计算库，前端依赖Node生态与Vite构建链。AI能力采用本地模型推理路径，能够在离线或弱网条件下保持核心功能可用。该部署策略兼顾了演示场景的稳定性与商业化场景的数据主权需求，尤其适用于对隐私与本地化有要求的医疗健康相关合作环境。

## 风险控制与演进路线

从技术风险角度看，当前系统已通过模块解耦降低了单点失效影响，但在大规模并发、跨设备采集一致性和模型长期漂移方面仍需持续优化。后续演进可围绕三条主线展开：其一，完善数据质量分级机制并引入自动告警；其二，扩展更多形态学特征与时频联合特征；其三，构建标准化评测基线以量化不同模型版本的收益与边界。通过上述演进，系统可从“可演示原型”逐步过渡为“可交付产品”。
